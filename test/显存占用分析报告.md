# Qwen2.5-7B LoRA 微调显存占用分析报告

## 1. 实验目的

本报告基于 `diary_homo/test/train.py` 的显存调试版，对当前项目在 `Qwen2.5-7B-Instruct + LoRA` 微调链路上的显存占用进行模块化分析，重点回答以下问题：

1. 各模块分别占了多少显存。
2. `4bit` 与 `8bit` 量化对显存的实际影响有多大。
3. LoRA 本身大约消耗多少显存。
4. `batch_size` 与 `token 长度` 增大时，显存大致会如何变化。

## 2. 实验设置

### 2.1 模型与数据

- 基座模型：`data/large_language_model/Qwen/Qwen2.5-7B-Instruct`
- 数据集：`data/dataset/json/boa_256.jsonl`
- 训练方式：LoRA 微调
- 保存方式：`save_steps=1`，确保第 1 步真实保存 `step_1`

### 2.2 小规模烟雾参数

- `max_length=64`
- `batch_size=1`
- `gradient_accumulation_steps=1`
- `epochs=1`
- `max_steps=1`
- `num_workers=0`
- `pin_memory=false`
- `shuffle=false`

### 2.3 输出目录

- 4bit debug 输出：`checkpoint/test_runs/train_memory_debug_4bit`
- 8bit debug 输出：`checkpoint/test_runs/train_memory_debug_8bit`

两组实验都真实完成了 1 个优化步，并成功生成：

- `step_1/adapter_model.safetensors`
- `step_1/adapter_config.json`
- `metrics_images/metrics_up_to_step_1.png`
- `memory_trace.jsonl`
- `memory_summary.json`

## 3. 数据来源说明

显存数据来自两套指标：

1. `torch.cuda.memory_allocated / reserved / max_memory_allocated / max_memory_reserved`
2. `nvidia-smi` 中当前训练进程显存与整卡显存

两者含义不同：

- `torch allocated` 更接近“活跃张量”占用。
- `torch reserved` 包含 PyTorch allocator 预留缓存。
- `nvidia-smi` 进程显存更接近系统视角下该进程对 GPU 的总占用。

因此本报告同时保留两套数字，不把它们混为一谈。

## 4. 核心结果

### 4.1 峰值结果总览

| 指标 | 4bit | 8bit | 8bit - 4bit |
|---|---:|---:|---:|
| `torch_peak_allocated` 峰值 | 9014.46 MiB | 12020.32 MiB | +3005.86 MiB |
| `torch_reserved` 峰值 | 11604.00 MiB | 13624.00 MiB | +2020.00 MiB |
| 训练进程显存峰值 | 11924 MiB | 13954 MiB | +2030 MiB |
| 整卡显存峰值 | 16497 MiB | 18527 MiB | +2030 MiB |

结论：

- 在本次完全同参数对照下，`8bit` 比 `4bit` 多占约 `2030 MiB`。
- `8bit` 的整卡峰值是 `18527 MiB`，仍然在你要求的 `19G` 以内。
- 量化的收益主要体现在“模型常驻显存”，不是体现在训练步中的激活开销。

## 5. 分模块显存占用

下表使用“训练进程显存”与“torch allocated”两套指标观察关键阶段。

### 5.1 4bit 分阶段

| 阶段 | 训练进程显存 | 相对起点增量 | torch allocated | 说明 |
|---|---:|---:|---:|---|
| `train_start` | 254 MiB | 0 MiB | 0.00 MiB | 调试脚本刚启动 |
| `after_base_model_load` | 7106 MiB | +6852 MiB | 5300.25 MiB | 4bit 基座模型载入 |
| `after_kbit_prepare` | 11266 MiB | +11012 MiB | 7381.89 MiB | k-bit 训练准备，显存大幅上涨 |
| `after_lora_injection` | 11392 MiB | +11138 MiB | 7690.70 MiB | LoRA 参数注入 |
| `after_first_forward` | 11484 MiB | +11230 MiB | 8839.02 MiB | 首个前向传播，活跃张量峰值出现 |
| `after_first_backward` | 11554 MiB | +11300 MiB | 8056.39 MiB | 反向传播后，激活部分释放 |
| `after_first_optimizer_step` | 11924 MiB | +11670 MiB | 8363.89 MiB | 第一次参数更新后达到进程峰值 |

### 5.2 8bit 分阶段

| 阶段 | 训练进程显存 | 相对起点增量 | torch allocated | 说明 |
|---|---:|---:|---:|---|
| `train_start` | 254 MiB | 0 MiB | 0.00 MiB | 调试脚本刚启动 |
| `after_base_model_load` | 9126 MiB | +8872 MiB | 8308.44 MiB | 8bit 基座模型载入 |
| `after_kbit_prepare` | 13288 MiB | +13034 MiB | 10389.58 MiB | k-bit 训练准备，显存继续上升 |
| `after_lora_injection` | 13414 MiB | +13160 MiB | 10697.83 MiB | LoRA 参数注入 |
| `after_first_forward` | 13512 MiB | +13258 MiB | 11845.35 MiB | 首个前向传播，活跃张量峰值出现 |
| `after_first_backward` | 13586 MiB | +13332 MiB | 11061.28 MiB | 反向传播后，激活部分释放 |
| `after_first_optimizer_step` | 13954 MiB | +13700 MiB | 11369.22 MiB | 第一次参数更新后达到进程峰值 |

### 5.3 模块拆解结论

对比两组增量可以得到很清楚的结论：

1. 数据集、tokenizer、dataloader 阶段几乎不占 GPU 显存。
2. 最大块的常驻显存来自“基座模型加载 + k-bit 训练准备”。
3. `4bit/8bit` 的主要差异发生在“模型常驻部分”，不是 LoRA 本身，也不是首个 batch 的激活。
4. LoRA 注入的额外显存，在 4bit 和 8bit 两组中几乎相同。
5. 首次 `forward/backward` 带来的动态开销在 4bit 与 8bit 中也几乎一致。

## 6. 4bit 与 8bit 的真实差异

### 6.1 量化差异主要发生在哪里

从同阶段对比看：

| 阶段 | 8bit - 4bit 训练进程显存差值 |
|---|---:|
| `after_base_model_load` | +2020 MiB |
| `after_kbit_prepare` | +2022 MiB |
| `after_lora_injection` | +2022 MiB |
| `after_first_forward` | +2028 MiB |
| `after_first_backward` | +2032 MiB |
| `after_first_optimizer_step` | +2030 MiB |

这说明：

- `4bit` 的节省几乎是“一条平移下来的固定差值”。
- 真正被省掉的是“模型权重相关的常驻占用”。
- 训练步中的激活、LoRA 梯度、优化器状态等动态部分并没有因为 4bit 而同比例缩小。

### 6.2 为什么不是减半

这是正常现象，原因如下：

1. 这里只是 `8bit -> 4bit`，不是 `fp16 -> 4bit`。
2. 量化主要作用于冻结的基座权重，不作用于全部训练过程中的所有张量。
3. LoRA 参数、梯度、优化器状态、激活值、workspace、allocator cache 仍然占据大量显存。
4. 因此 4bit 节省的是“固定底座”，不是把整个训练进程显存按 1/2 压缩。

## 7. LoRA 显存估算

### 7.1 实测 trainable 参数规模

两组实验里，LoRA 可训练参数数目相同：

- `trainable = 80,740,352`

这个数字是可信的；而 `total` 参数数目在 4bit 和 8bit 下不同，不应拿来比较逻辑模型大小，因为 bitsandbytes 的量化层内部参数表示方式不同。

### 7.2 理论估算

如果 LoRA 参数按 `fp32` 存储：

- 参数本体：`80,740,352 * 4 bytes ≈ 307.99 MiB`
- 梯度：约 `307.99 MiB`
- AdamW 两组状态 `m/v`：约 `615.98 MiB`

理论上，这部分长期/阶段性相关显存大约是：

- 参数本体：`~308 MiB`
- 参数 + 梯度：`~616 MiB`
- 参数 + 梯度 + AdamW 状态：`~1232 MiB`

### 7.3 与实测的对应关系

实测中：

- 4bit 从 `after_kbit_prepare` 到 `after_lora_injection`：`torch allocated +308.81 MiB`
- 8bit 从 `after_kbit_prepare` 到 `after_lora_injection`：`torch allocated +308.25 MiB`

这非常接近 LoRA 参数本体的 `~308 MiB` 理论值。

进一步看：

- 4bit 从 `after_lora_injection` 到 `after_first_optimizer_step`：`torch allocated +673.19 MiB`
- 8bit 从 `after_lora_injection` 到 `after_first_optimizer_step`：`torch allocated +671.39 MiB`

这说明第一次优化步后，LoRA 相关的梯度和优化器状态开始真正进入稳定显存占用区间。

结论：

1. LoRA 本体大约就是 `~308 MiB`。
2. 真正进入训练后，LoRA 训练相关的总额外占用会继续往上加到 `~0.7 GiB` 左右的稳定区间。
3. 若把参数、梯度、优化器状态全都按最保守方式估算，`~1.2 GiB` 量级是合理的上界判断。

## 8. batch 与 token 长度对显存的粗略估算

### 8.1 从本次实验能直接观察到的量

在两组实验中：

- 4bit：`before_first_forward -> after_first_forward` 的 `torch allocated` 增量是 `+1148.32 MiB`
- 8bit：`before_first_forward -> after_first_forward` 的 `torch allocated` 增量是 `+1147.52 MiB`

这两个数字几乎完全一致，说明：

1. `forward` 阶段的动态显存主要由激活决定。
2. 它与量化模式关系不大。
3. 它更接近和 `batch_size * seq_len` 成正比。

### 8.2 一阶工程估算公式

以当前实验点 `(batch_size=1, seq_len=64)` 作为基准，可做一个非常粗略的一阶估算：

```text
forward 激活动态增量 ≈ 1.15 GiB * (batch_size * seq_len / 64)
```

注意：

- 这是“同模型、同 LoRA、同训练脚本、同单卡、同量化策略”下的工程估算。
- 它适合快速预估，不适合当精确容量规划公式。
- 真正的总显存还要叠加固定底座、optimizer 状态、allocator cache、CUDA workspace。

### 8.3 按 `torch allocated` 的粗略峰值估算

当前实验的“前向前固定底座”约为：

- 4bit：`7690.70 MiB`
- 8bit：`10697.83 MiB`

则前向峰值的粗略估算如下：

| 配置 | 4bit 估算前向峰值 | 8bit 估算前向峰值 |
|---|---:|---:|
| `B=1, L=64` | `~8839 MiB` | `~11845 MiB` |
| `B=1, L=128` | `~9988 MiB` | `~12993 MiB` |
| `B=2, L=64` | `~9988 MiB` | `~12993 MiB` |
| `B=1, L=256` | `~12286 MiB` | `~15291 MiB` |
| `B=2, L=128` | `~12286 MiB` | `~15291 MiB` |

这些数字表示的是 `torch allocated` 的粗略趋势，不是 `nvidia-smi` 整卡最终读数。

### 8.4 参数影响排序

在这条训练链上，显存影响通常可以这样理解：

1. `quantization_mode`：主要影响固定底座，占用差异最稳定。
2. `lora_r`：主要影响 LoRA 参数、梯度、优化器状态，近似线性增长。
3. `batch_size`：主要放大激活显存，通常近似线性。
4. `max_length`：同样主要放大激活显存，通常近似线性。
5. `gradient_accumulation_steps`：对单步峰值影响远小于直接增大 `batch_size`，但会影响训练时间与累计更新逻辑。

## 9. 结论

本次调试实验可以得出以下结论：

1. 当前项目在 `Qwen2.5-7B + LoRA` 小规模微调中，GPU 显存的绝对大头是“基座模型常驻 + k-bit 准备”。
2. 数据集、tokenizer、dataloader 几乎不占 GPU 显存。
3. LoRA 本体大约占 `~308 MiB`，真正训练后连同梯度/状态进入 `~0.7 GiB` 左右的稳定区间，理论上保守上界可按 `~1.2 GiB` 估。
4. `4bit` 相比 `8bit`，在本次同参数对照中稳定节省了约 `2030 MiB`。
5. `8bit` debug 实验整卡峰值为 `18527 MiB`，已经满足“不超过太大显存、控制在 19G 之内”的要求。
6. `batch_size` 与 `token 长度` 的影响主要体现在激活显存，它们对动态峰值的影响通常比 LoRA 本体更直接。

## 10. 建议

如果后续目标是继续压显存，建议优先顺序如下：

1. 保持 `4bit`
2. 优先减小 `max_length`
3. 再减小 `batch_size`
4. 若还不够，再考虑降低 `lora_r`

如果后续目标是继续做更精确的容量规划，建议追加两类实验：

1. 固定 `batch_size=1`，跑 `max_length=64/128/256`
2. 固定 `max_length=64`，跑 `batch_size=1/2/4`

这样就能把“token 长度斜率”和“batch 斜率”从估算提升为真实测量曲线。
